# Provisioning request — Magna Read-Across v2 (prod)

## In plain English — why we're asking

**What we have today.** The Magna Read-Across dashboard runs as a single
website that reads its data from a folder of static JSON files in cloud storage
(blob). Whenever the data changes, someone has to regenerate those files and
re-upload them. The website also holds a shared key to reach that storage,
which is the equivalent of a password sitting in a config file.

**What's changing.** The new version of the app keeps the same dashboard but
puts the data in a proper database. Instead of files being regenerated by
hand, the database is updated through normal data pipelines and the dashboard
reads it live.

**Why this matters.**

- **Better data.** Filters, rankings, and benchmarks are computed in the
  database, so the dashboard always reflects the latest numbers — no waiting
  for a file rebuild.
- **Stronger security.** The new app uses *Managed Identity*, which is the
  cloud equivalent of an employee badge instead of a password. The app
  identifies itself to the database by who it is, not by carrying a secret.
  No shared key, no password to leak or rotate.
- **Easier to maintain.** The team can update content (videos, slide decks,
  recommendations) by inserting database rows — same process used by every
  other internal app — instead of editing JSON files.
- **Lines up with firm standards.** The new version uses the same Azure +
  Managed Identity pattern that other internal apps use, so it inherits the
  firm's security review, monitoring, and on-call playbooks for free.

**Data status (OneStream → SQL).** OneStream remains the upstream system of
record for P&L and Wave data and is already connected and pushing data on its
existing schedule. Wiring those feeds directly into the new SQL database is
**in progress and expected to take another 3–4 weeks**. Until that pipeline is
live, the new app runs against representative seed data already loaded into
the schema, so the platform set-up, security review, and end-user testing
aren't blocked while data engineering finishes the integration.

**What we need from the platform team.** A small set of Azure resources for
the new app to live in: somewhere to run the website and its API, a database,
an identity for the app to use, and the usual logging/monitoring plumbing. The
existing app keeps running unchanged until the new one is verified — there is
no risk to today's users while we set this up.

**Cost.** Roughly **$60–140 per month** for production, depending on traffic.
The dashboard is internal and lightly used, so the database can sleep when
nobody is on it.

**Timing.** We'd like the resources stood up in the next sprint so we can
deploy and validate against real users before retiring the existing app.

## 1. Why we are filing this

The existing `magna-readacross` app is a single static Nginx container fed from
blob storage. The replacement, `magna-readacross-v2`, is a two-tier app
(.NET 8 API + Angular SPA) that reads from a relational `readacross` schema
in Azure SQL. The API authenticates to the database with **Managed Identity**
— no SQL username/password, no blob, no shared key.

We need Azure resources provisioned so we can stand up v2 in **prod** alongside
the existing app. The two apps will run in parallel until cutover; this ticket
covers v2 only.

## 4. Identity → SQL wiring (the part that replaces the blob key)

After (9) and (10/11) exist, run **once** against the SQL database (any
account with `db_owner` works for this bootstrap step):

```sql
-- Replace <uami-name> with the User-Assigned Managed Identity from item 9.
CREATE USER [<uami-name>] FROM EXTERNAL PROVIDER;

ALTER ROLE db_datareader ADD MEMBER [<uami-name>];
ALTER ROLE db_datawriter ADD MEMBER [<uami-name>];

-- Optional: schema-bound permissions instead of the broad role.
GRANT SELECT, INSERT, UPDATE, DELETE ON SCHEMA::readacross TO [<uami-name>];
```

The .NET API picks up the identity automatically via `DefaultAzureCredential`
(see `api/Program.cs`) — no app-side change needed once the role assignment is
in place.

> **No SQL admin user needs to be created for the app.** The Entra admin on
> the SQL server (item 10) is only used for one-time bootstrapping and break-glass.

## 5. CI/CD considerations (for the platform team)

The repo (`magna-readacross-v2/`) is ready for a build → push → revision-update
flow:

1. Build images (`api/Dockerfile`, `web/Dockerfile`).
2. Push to the registry (item 7) tagged with the commit SHA.
3. `az containerapp update --image …` for both Container Apps.

Recommended GitHub OIDC federation:

- One **federated credential** per environment on a deployer App Registration
  (or a separate UAMI), with `Contributor` on the resource group (item 1).
- No long-lived service-principal secrets stored in GitHub.
